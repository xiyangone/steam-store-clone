# 提示词设计意图说明

对应提示词文件：

- [STEAM_REBUILD_PROMPT.md](Z:/1/STEAM_REBUILD_PROMPT.md)

## 设计目标

这份提示词不是单纯让模型“生成一个像 Steam 的页面”，而是刻意把任务拆成了「视觉还原 + 工程落地 + 自动部署」三个层面，目的是让产出更稳定、更接近真实可交付项目。

## 为什么这样设计

### 1. 先锁定技术边界

提示词里明确要求继续使用 `React + TypeScript + Vite`，是为了避免模型自行切换技术栈，导致项目结构、构建配置和已有代码资产失去复用价值。

### 2. 强调“在现有结构上重构”

这一条的目的，是约束模型优先理解当前项目，再在原组件和数据层基础上迭代，而不是推倒重来。这样更适合真实协作，也更利于后续维护。

### 3. 把视觉还原拆成可执行的区域要求

提示词分别点名了：

- Hero 视频区
- 排行榜区
- 页脚区
- 按钮、文字、间距、响应式

这样做的原因，是 Steam 首页的相似度主要体现在这些高识别度区域。把重点拆开写，能减少模型只做“整体像，但细节不像”的情况。

### 4. 同时要求样式和内容接近官网

如果只要求“样式像”，模型容易保留占位内容；如果只要求“内容像”，页面结构又可能不对。所以提示词同时约束内容、布局和视觉节奏，提升复刻完整度。

### 5. 明确数据应收敛到 mock / 配置层

这一点是为了避免把图片、标题、价格、截图等内容散落在多个组件里。数据集中后，后续替换官网素材、做多语言、改榜单内容都会更轻松。

### 6. 强制要求“先定位根因，再修复”

这是为了避免模型遇到 bug 时只做表面修补。像这次 GitHub Pages 图片 404，本质就是部署子路径和静态资源路径的关系，如果不强调根因，模型很容易做出临时但不稳的修法。

### 7. 把验证写进提示词

提示词明确要求至少做构建验证，涉及页面行为时还要做浏览器检查。这样可以显著降低“代码写完但线上不能用”的概率。

### 8. 把自动部署作为目标的一部分

很多生成式提示词只覆盖开发，不覆盖部署，结果就是本地能跑、线上不能预览。这份提示词把 GitHub Pages 自动部署写成硬要求，是为了保证最终成果一定有线上地址。

## 这份提示词的预期效果

理想情况下，使用这份提示词后，模型不应该只输出一段静态代码，而应该完成下面这条闭环：

1. 理解现有项目结构
2. 修改代码还原 Steam 官网关键区域
3. 修复与部署相关的路径、资源和配置问题
4. 配置 GitHub Actions 自动发布
5. 给出最终线上预览地址

## 适用场景

这份提示词适合：

- 在现有 Steam 复刻项目上继续精修
- 需要“代码 + UI + 部署”一体化产出
- 希望模型少做泛泛建议，多做真实交付

它不适合：

- 只想要一张静态设计稿
- 只想讨论思路，不想改代码
- 完全脱离当前仓库、从零换技术栈重做
